# Contributing to OCHA AI Prototypes

Thank you for your interest in contributing.

This repository exists to support **safe experimentation and learning** through AI‑assisted prototypes and visualisations. Contributions are welcome — as long as they follow the governance and transparency rules described below.

---

## What this repository is for

This repository hosts **Tier 1 – Shareable Prototypes** under OCHA’s *AI Vibe Coding Guidance*.

Appropriate contributions include:
- Prototype dashboards
- Interactive visualisations
- Exploratory HTML / JavaScript web apps
- Demonstrations created using AI‑assisted “vibe coding”

All content here must be clearly labelled as **prototype / exploratory** and must **not** be used for operational or decision‑making purposes.

---

## What does NOT belong here

Please do **not** contribute:
- Operational or decision‑support tools
- Production or public‑facing systems
- Validated or authoritative figures
- Artefacts using sensitive, personal, or restricted data
- Anything intended to directly influence decisions

If your work starts to move in this direction, it likely falls under **Tier 2** and should be discussed before being shared here.

---

## How to contribute your prototype

### 1. Contact us
- Reach out to the OCHA Technology Services Section who will grant you access to this repository.

---

### 2. Basic setup
- Create a new folder under `/docs/` using a short, descriptive name
- Your folder should include at minimum:
  a) `index.html` (the prototype)
  b) Any required JS / CSS files

A small `README.md` inside the prototype folder is encouraged, especially if the logic is not obvious.

---

### 3. Mandatory transparency (required)

Every prototype **must clearly display or document**:

- **Owner** – name and role
- **Purpose** – what the prototype is and is *not* intended to do
- **Data sources** – public, synthetic, or sample data only
- **AI involvement** – what was generated or assisted by AI
- **Limitations** – known gaps, assumptions, or risks

This information can appear:
- Directly in the UI, and/or
- In a README inside the prototype folder

If this information is missing, the prototype is **not ready to be shared**.

---

### 4. Open a Pull Request

Open a Pull Request against the `main` branch.

Maintainers may request:
- Clearer labelling
- Additional caveats
- Adjustments to framing
- Removal of inappropriate data or claims

This is part of keeping the repository safe and useful for everyone.

---

## Review and oversight

This repository follows the rules in `GOVERNANCE.md`.

Maintainers may:
- Request changes before merging
- Pause or remove prototypes that drift toward Tier 2 use
- Recommend escalation or relocation when appropriate

Governance is intended to **support safe innovation**, not block it.

---

## When in doubt

If you are unsure whether your prototype belongs here:
- Assume the higher‑risk tier
- Raise the question early
- Ask in the relevant AI or data community of practice

It is always easier to adjust *before* something is widely shared.
